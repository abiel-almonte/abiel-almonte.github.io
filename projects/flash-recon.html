Fig. 1 Reconstructions on TartanAir sequences (ME003, MH004, MH006)

Flash-Recon runs dense monocular SLAM and Gaussian Splatting jointly, producing live 3D reconstructions on a single consumer GPU. The system is designed performance-first, the core optimization targets visual odometry — [photon -> pose] — since it runs per-frame and gates everything downstream. Fast odometry means Gaussian Splatting updates stream to the viewer asynchronously, keeping the full pipeline real-time.

How it works

Flash-Recon reimplements DROID-SLAM from the ground up. This means, Lie group operations, bundle adjustment, factor graphs, and system orchestration.

The system runs three threads on a single GPU, communicating through lock-free queues and read-only snapshots:

  1. Slammer — processes every frame through a learned feature extractor, computes optical flow between frames, and solves bundle adjustment to refine camera poses and depth for keyframes.
  2. Splatter — picks up new poses, backprojects their depth maps into 3D Gaussians, and optimizes them with differentiable rasterization. Deforming the previous gaussians inplace.
  3. Viewer — streams Gaussian snapshots over WebSocket to a browser, where WebGL renders the reconstruction live. 32 bytes per Gaussian, packed binary.

Rasterization inherited from nerfstudio-project/gsplat. Viewer modifies antimatter15/splat.

The splatter yields GPU time to the slammer when both are active, so SLAM accuracy is never sacrificed for prettier Gaussians.

Where the speed comes from

One of DROID-SLAM's main bottlenecks is bundle adjustment, the process of solving a sparse linear system over the pose graph every time a new keyframe arrives. The original runs this across two 3090s.

Flash-Recon rewrites the BA solver as a custom CUDA kernel using Schur complement decomposition, bringing the per-call cost from 4.98 ms down to 1.34 ms on EuRoC.

The rest is system design. Three threads share a single GPU through lock-free queues and read-only snapshots. No mutexes, no multi-GPU synchronization. When SLAM refines poses, Gaussians deform in-place to stay consistent avoiding re-initialization.

Is there more headroom?

Yes, there might! The correlation kernel that computes the correlation volume for RAFT to estimate optical flow is inherited from DROID-SLAM and is the next target for optimization.

Ideally we can exploit the assu/cmption that optical flow is spatially smooth. So, neighboring pixels have similar flow vectors, which means neighboring threads need overlapping image patches.

With this setup we can cooperatively load image patches instead of loading each patch independently, cutting global memory traffic significantly.
